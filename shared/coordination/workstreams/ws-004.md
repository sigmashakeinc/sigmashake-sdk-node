# ws-004: Production Auto-Recovery & Self-Healing System

**Status:** in-progress
**Created:** 2026-03-15
**SLA Target:** 99.9% (≤43 min downtime/month)

## Goal

Deploy-time auto-rollback, independent watchdog monitoring, self-healing remediation, and public status page. Eliminate undetected bad deploys as the #1 cause of downtime.

## Architecture Decisions

| Question | Decision | Rationale |
|----------|----------|-----------|
| Watchdog location | New crate `crates/watchdog/`, 8th CF Worker | Same tooling, independent deploy, zero deps on other workers |
| Monitor hierarchy | CF Health Checks → watchdog → soc-core | No single point of failure |
| Auto-rollback window | Within 5 min of deploy only | Clear causal link for deploy-time failures |
| Deploy strategy | Sequential staged deploy with health gates | CF Workers don't support native blue-green |

## Contracts

- `GET /health` → `HealthResponse { status, version }`
- `GET /v1/health/deep` → `DeepHealthResponse { status, version, checks[], checked_at }`
- `GET /v1/watchdog/status` → full status of all monitored workers (JWT)
- `GET /v1/watchdog/history` → recent health check history (JWT)
- `GET /v1/status` → public status page data (no auth, CORS)
- `GET /v1/webhooks/dlq` → dead letter queue entries (JWT)
- `POST /v1/webhooks/dlq/:id/retry` → retry a failed webhook (JWT)
